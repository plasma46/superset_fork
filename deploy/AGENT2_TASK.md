# Задача

## Что нужно реализовать

В чарте **Table V2** (`plugin-chart-ag-grid-table`) добавить функционал редактируемых
"комментариев" — пользователь вводит значения прямо в строки таблицы и сохраняет их
через REST API (бэкенд уже будет готов другим агентом по контракту ниже).

## 1. Control Panel (controlPanel.tsx в этом плагине)

Добавь секцию "Comments" с полями:
- `enabled` (boolean) — включить функционал
- `database_id` (select существующих Database connections Superset — используй
  стандартный `DatasourcePanelDropdown`/датасорс-селектор, который уже есть в других
  контролах Superset, не изобретай свой)
- `schema`, `table` (текстовые поля)
- `key_mapping` — список пар {view_column: строка, target_column: строка},
  динамически добавляемые строки (как массив объектов)
- `fields` — список объектов:
  ```ts
  {
    view_column: string;
    target_column: string;
    type: 'text' | 'number' | 'dropdown_static' | 'dropdown_dynamic';
    options?: { label: string; value: any }[];       // для dropdown_static
    dataset_id?: number;                               // для dropdown_dynamic
    value_column?: string;
    label_column?: string;
  }
  ```
- `refresh_chart_id` (number, опционально) — ID чарта для force refresh после сохранения

Всё это сохраняется в `form_data.comment_config` — именно под этим ключом,
backend будет читать оттуда же.

## 2. Рендер таблицы

- Добавь колонку с чекбоксами слева (для bulk-select), используя стандартный
  ag-grid checkbox column (`checkboxSelection: true` на первой колонке)
- Для каждого поля из `fields` — рендери ячейку как:
  - `text` → обычный текстовый input
  - `number` → input с `type="number"`, **запрети ввод нечисловых символов** до сохранения
    (валидация на клиенте — `onKeyPress`/regex, плюс подсветка ошибки если значение невалидно)
  - `dropdown_static` → select с опциями из `options`
  - `dropdown_dynamic` → select, данные грузи **один раз при монтировании компонента**
    через существующий API `/api/v1/dataset/{id}/data/` или аналогичный механизм
    получения данных датасета (посмотри как это делается в `NativeFilters` для
    динамических фильтров — там есть похожий паттерн загрузки опций из датасета),
    закэшируй в локальном state, не перезапрашивай на каждый рендер.

## 3. Локальный state изменений (dirty tracking)

- Храни несохранённые изменения в локальном React state компонента,
  ключ — композитный ключ строки (сериализованные значения из `key_mapping`)
- Одна строка может содержать изменения по нескольким полям одновременно

## 4. Bulk-режим ("Mass input")

- Когда выбрано ≥2 строк через чекбоксы — показывай кнопку/тулбар "Mass input"
- Клик → модалка: выбрать поле из `fields` → ввести одно значение (с той же
  валидацией по типу) → "Apply" → значение проставляется во все выбранные строки
  в локальном dirty-state (не отправляется на сервер сразу)

## 5. Кнопка Save

- Одна кнопка "Сохранить", активна когда есть хоть одна dirty-строка
- При клике собери payload по контракту:
  ```json
  {
    "records": [
      { "keys": {...}, "fields": {...}, "is_delete": false }
    ]
  }
  ```
- POST на `/api/v1/chart/{chart.id}/comments` (используй стандартный
  `SupersetClient` из `@superset-ui/core` для запроса, он уже добавляет CSRF-токен)
- При успехе:
  - очисти dirty-state
  - покажи toast "Сохранено" (используй существующий механизм toast/notification Superset)
  - если `comment_config.refresh_chart_id` задан — задиспатчь:
    ```ts
    import { refreshChart } from 'src/components/Chart/chartAction';
    dispatch(refreshChart(refresh_chart_id, true, dashboardId));
    ```
    (dashboardId доступен через пропсы/контекст чарта — посмотри как другие компоненты
    дашборда его получают, например через `useSelector` на `dashboardInfo.id`)
- При ошибке (400/403/422) — покажи понятную ошибку пользователю, не очищай dirty-state

## 6. Кнопка "Удалить" на строке

- Тот же механизм Save, но отправляет ОДНУ запись с `is_delete: true`
  и текущими значениями `keys` (fields можно не отправлять или отправить пустыми —
  уточни у backend-агента финальный формат если будет неоднозначность)

## 7. Тесты (Jest)

Покрой:
- Валидацию числового поля (отклоняет текст)
- Сборку payload для одной строки и для bulk-режима
- Логику применения "Mass input" значения к нескольким выбранным строкам
- Очистку dirty-state после успешного save

Не нужно: Cypress/e2e — не в этой итерации.

## Важно

- Это новая функциональность поверх существующего ag-grid Table плагина — НЕ переписывай
  существующий рендеринг таблицы, только дополняй его.
- Не трогай файлы вне `superset-frontend/plugins/plugin-chart-ag-grid-table/`.
- Если бэкенд-эндпоинт `/api/v1/chart/{id}/comments` ещё не готов в момент твоей работы —
  пиши и тестируй фронтенд с mock-ответами (jest.mock на SupersetClient), интеграцию
  проверим вместе после того как оба агента закончат.
